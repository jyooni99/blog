2025년 9월부터 11월 4일까지, 총 8주간 진행된 **코드잇 단기 심화 프로젝트** <br/>
그리고 12월 1일부터 1월 18일까지 4주간 이어진 **코드잇 커리어 프로그램**까지 무사히 수료하게 되었다.

이번 회고를 통해 구름 부트캠프 수료 이후에 또 단기 심화 프로젝트를 듣게 된 이유, 그리고 이번 프로젝트에서 배운 점들을 정리해보려고 한다.

## 단기 심화를 듣게 된 이유

단기 심화를 듣게 된 이유는 **내가 아직 많이 부족하다고 느꼈기 때문**이다.

구름 딥다이브 부트캠프에서 우수 수료생으로 선발되었고, 그 덕에 인턴십 지원 기회를 얻을 수 있었다.
지원하는 기업의 블로그 글과 과제 테스트 설명을 읽어보니 이곳은 **구현보다 왜 이렇게 설계했는지**를 중요하게 생각하는 것처럼 보였고, 그래서 사소한 것들도 왜 이렇게 설계했는지 고민한 흔적들을 `ReadMe`에 정리해 보여주었다.

<MdxImage src="/posts/code-it-review/readme-example.png" alt="readme-example" />

다행히 과제 테스트를 통과할 수 있었다.

하지만, 면접 과정에서 질문에 대한 대답은 커녕, 질문 조차 이해하지 못하는 나를 발견하면서
한 없이 작아지는 기분이 들었고, **내가 아직 부족하다는 사실을 체감하게 되었다.**

특히 React와 Next.js를 사용하면서도, 정작 **왜 이 기술을 사용해야 하는지**에 대해 모르고 있다는 것을 깨달았다.
그냥 부트캠프에서 배우니까, 좋겠거니 쓰고 있던 것이다. 과제 테스트에서 `Jest`나 `React Testing Library` 같은 테스트 도구를 사용해야 했는데,
해당 도구들을 사용해본 경험이 없어 테스트 코드를 충분히 작성하지 못했던 점도 아쉬움으로 남았다.

자신감이 떨어진 채로 면접을 보다 보니, 다른 기업들도 줄줄이 탈락하게 되었고,
나의 부족함을 채우고자 코드잇 단기 심화 프로젝트를 선택하게 되었다.

## 지원 과정

해당 과정은 기술 역량 검증을 위한 과제 테스트와 인터뷰 과정을 거쳐 선발되는 방식이었는데,
서류 지원 시 제출한 포트폴리오 결과에 따라 테스트가 면제되기도 한다.

나같은 경우는 포트폴리오를 제출해 과제 테스트를 면제받고 인터뷰를 보게 되었다.

인터뷰는 비대면 방식으로 진행된다. Zoom이나 Google Meet 같은 플랫폼을 사용하는 방식은 아니었고,
질문에 답변을 하면 답변 내용이 녹화되어 제출되는 방식이기 때문에 긴장감은 조금 덜했던 것 같다.
재답변 기회가 한 번 있긴 하지만, 주어진 시간이 지나버리면 답변 기회가 사라지기 때문에 시간을 체크하는 것이 중요했다.

예상치 못한 방식에 답변을 조금 절긴 했으나.. 아무튼 **최종 합격을 받았게 되었다!**

## 학습 내용

8주 중 2주는 현업에서 사용하는 기술들을 학습하는 이론 과정이었다.

- **TailwindCSS**: 유틸리티 클래스 기반 스타일링
- **CSS 애니메이션**: transition, @keyframes 활용, Framer Motion, intersection-observer 활용
- **웹 브라우저**: 로컬 스토리지와 세션 스토리지, 쿠키
- **테스트 라이브러리**: Jest, React Testing Library
- **CI/CD**: GitHub Actions, Vercel 등

개인적으로 CSS 애니메이션과 테스트 라이브러리가 가장 재밌었던 것 같다.

## Zipkok 프로젝트

<MdxImage src="/posts/code-it-review/zip-kok.png" alt="zip-kok-project-image" />

Zipkok 프로젝트는 오프라인 만남이 부담스럽거나 바쁜 일정 속에서 사람들과 소통하고 싶은 사람들을 위해 온라인에서 편안하게 참여할 수 있도록 기획된 모임 서비스이다.
프론트엔드 3명, 백엔드 2명, 디자이너 1명으로 구성된 팀이고, Next.js와 TailwindCSS, Tanstack Query, Zustand를 사용해 개발했다.

- [Zipkok 프로젝트 배포 링크](https://zipkok.vercel.app/)

### 백엔드와의 협업

이전 프로젝트에서는 요구사항 명세서를 기반으로 백엔드에서 API를 먼저 개발하고, 프론트엔드에서는 완성된 API를 호출해 화면에 데이터를 표시하는 방식으로 작업을 진행했다.
구조 자체는 익숙한 협업 방식이었지만, 백엔드에서 구현한 기능이 프론트엔드에서 기대하던 동작 방식과 달라 여러 차례 수정이 오가야 했다.

<Callout theme="zinc">
**QR 명함 업데이트 기능**

한 사용자가 자신의 QR 명함을 업데이트하면 해당 명함을 저장한 다른 사용자들의 화면에도 변경 사항이 반영되어야 한다.
하지만 자신의 명함만 업데이트 될 뿐, 다른 사용자들의 명함은 업데이트 되지 않았다.
DB가 참조 형태가 아니라 복사 형태로 구현되어 있었기 때문이다.

이 사실을 너무 늦게 알았기 때문에, 구조를 다시 설계하기에는 현실적으로 어려운 상황이었고, 해당 기능은 구현하지 못한 채 프로젝트를 마무리해야 했다.

</Callout>

이전 프로젝트에서 아쉬웠던 점을 보완하기 위해 이번 프로젝트에서는 내가 생각하고 있는 필드명, 데이터 구조 등을 초기 단계에서 미리 공유했고,
기능의 동작 흐름을 설명했다. 그 덕분에 이전 프로젝트에서 겪었던 구조적인 문제를 초기 설계 단계에서 예방할 수 있었다.

### 학습했던 기술 적용

이번 프로젝트에서는 **MSW**를 활용해, 백엔드 API 구현 전에 프론트엔드에서 데이터를 화면에 표시할 수 있도록 구현했다.
덕분에 API 개발 전에 UI를 먼저 구현할 수 있어, 개발 속도를 높일 수 있었다.

프로젝트가 종료된 이후에는 팀원들과 함께 **Jest**와 **React Testing Library**를 도입해 테스트 코드를 작성해주었다.
주요 컴포넌트와 핵심 로직을 중심으로 테스트를 작성하여, 리팩토링을 했을 때 기존 기능이 깨지지 않는지 확인할 수 있도록 하였다.

MSW와 테스트 코드 도입으로 코드의 안정성은 높아졌지만, 또 다른 문제를 마주하게 되었다.

Props가 변경되거나 기능이 추가될 때마다 기존 테스트 코드에서 에러가 발생했고, 그 때마다 실제 기능 코드와 테스트 코드를 함께 수정해야 한다는 점이다.
안정성 대신 유지보수 비용이 증가하는 문제가 발생했고, 이를 해결하기 위한 방법을 찾아봐야할 것 같다.

## 커리어 프로그램

마지막으로 코드잇 심화 프로젝트를 수료하면, 4주 동안 진행되는 커리어 프로그램을 진행할 수 있는데, 이건 **정말정말 들어야 한다!!!**
보통은 현재 이력서에 대한 첨삭만 진행하는데, 이 프로그램에서는 이력서에 어필하고 싶은 부분을 어떻게 보여주면 좋을지,
프로젝트를 통해 얻은 경험을 어떻게 보여주면 좋은지를 피드백해주기 때문에 스스로 방향성을 잡을 수 있다!

실제 면접처럼 기술 모의 면접을 진행하고, 내 답변에 대한 피드백을 주고 받는 과정도 너무 좋았다 :)
